# ArkyHub Positioning Statement

Based on April Dunford's "Obviously Awesome" — Positioning Framework.

---

## The Framework

Dunford defines 5 components of positioning that must work together:

| Component | Question it answers |
|-----------|-------------------|
| **Competitive Alternatives** | What would customers use if ArkyHub didn't exist? |
| **Unique Attributes** | What does ArkyHub have that the alternatives don't? |
| **Value** | What benefit do those attributes deliver? |
| **Target Customer** | Who cares the most about that value? |
| **Market Category** | What context makes the value obvious? |

---

## Applied to ArkyHub

### 1. Competitive Alternatives

What architecture teams actually use today (not other SaaS — the real competition):

- Email + WhatsApp for sharing plans and updates
- Google Drive / Dropbox / OneDrive for storing project files
- WeTransfer for sending large files to contractors
- Informal folder structures with naming conventions like `Plan_v3_FINAL_revised2.pdf`
- Occasional use of Procore or BIM 360, but controlled by the contractor — not the architect

The real competitor is **"the patchwork"** — the combination of free/generic tools that sort of works but doesn't scale.

### 2. Unique Attributes

What ArkyHub has that the patchwork doesn't:

- **Plan versioning with issue tracking** — issues are tied to the exact plan version, not floating in a chat thread
- **BIM viewer in the browser** — no software installation, no licenses, anyone can view
- **Virtual tour embedding** — walkthroughs live inside the project, not in a separate link
- **Multi-company isolation** — each firm's data is fully separated, unlike a shared Drive folder
- **Role-based access per stakeholder type** — the contractor sees plans, the client sees the dashboard, the architect manages everything

### 3. Value (What those attributes actually deliver)

| Attribute | Value it creates |
|-----------|-----------------|
| Plan versioning + issue tracking | No more "which version is this?" — every issue points to the exact drawing |
| BIM viewer in browser | Stakeholders review 3D models without buying software or learning new tools |
| Virtual tour embedding | Clients understand project progress without visiting the site |
| Multi-company isolation | Work with multiple clients without data leaking between them |
| Role-based access | Share everything relevant, nothing more — contractors, clients, and team members each get their view |

### 4. Target Customer

**Architecture firms (5–50 people) managing multiple concurrent projects with external stakeholders.**

Specifically:

- Firms that already share documents digitally but lack a centralized system
- Firms where the architect — not the contractor — wants to control the documentation workflow
- Firms that work with clients who expect visibility into project progress

The decision maker is typically the **firm owner or principal architect**. The daily user is the **project manager or project architect**.

### 5. Market Category

**Project documentation platform for architecture firms.**

Not "construction management software" (that's Procore's territory — contractor-centric).
Not "file storage" (that's Google Drive — generic, no AEC context).
Not "BIM collaboration" (that's Autodesk's territory — too technical, too expensive).

ArkyHub sits in a space that doesn't have a dominant player yet: **architect-led project documentation with stakeholder access.**

---

## The Positioning Statement

> For **architecture firms managing multiple projects with external stakeholders** who are **tired of scattered documents, version confusion, and no single place to share project progress**, ArkyHub is a **project documentation platform** that **centralizes plans, BIM models, and virtual tours with version control and role-based access in one browser-based workspace**. Unlike **email + Google Drive + WhatsApp** (the current patchwork), ArkyHub **gives the architect full control of project documentation while providing every stakeholder — team members, contractors, and clients — exactly the access they need**.
